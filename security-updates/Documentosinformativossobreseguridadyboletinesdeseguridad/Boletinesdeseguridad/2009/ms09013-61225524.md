Security Bulletin

Boletín de seguridad de Microsoft MS09-013 - Crítica
====================================================

Vulnerabilidades en los servicios HTTP de Windows podrían permitir la ejecución remota de código (960803)
---------------------------------------------------------------------------------------------------------

Publicado: martes, 14 de abril de 2009 | Actualizado: miércoles, 29 de abril de 2009

**Versión:** 1.1

### Información general

#### Resumen ejecutivo

Esta actualización de seguridad resuelve una vulnerabilidad de la que se ha informado de forma pública y dos vulnerabilidades de las que se ha informado de forma privada en los servicios HTTP de Microsoft Windows (WinHTTP). La vulnerabilidad más grave podría permitir la ejecución de código remoto. Un atacante que aprovechara esta vulnerabilidad podría lograr el control completo de un sistema afectado. De esta forma, un intruso podría instalar programas; ver, cambiar o eliminar datos; o crear cuentas nuevas con todos los derechos de usuario. Por tanto, los usuarios cuyas cuentas estén configuradas con pocos derechos de usuario en el sistema correrían un riesgo menor que aquellos que cuenten con derechos de usuario administrativos.

Esta actualización de seguridad se considera crítica para todas las ediciones compatibles de Microsoft Windows 2000, Windows XP, Windows Server 2003, Windows Vista y Windows Server 2008. Para obtener más información, vea la subsección **Software afectado y no afectado**, en esta sección.

La actualización de seguridad corrige las vulnerabilidades al cambiar la forma en que los servicios HTTP de Windows tratan los errores y valida los certificados, y al garantizar que los servicios HTTP de Windows usan correctamente los mecanismos de protección de reflejo de credenciales NTLM. Para obtener más información acerca de las vulnerabilidades, consulte la subsección Preguntas más frecuentes (P+F) de la entrada de vulnerabilidad específica en la sección siguiente, **Información sobre la vulnerabilidad**.

**Recomendación.** La mayoría de los clientes tienen habilitada la actualización automática y no deben realizar ninguna acción porque esta actualización de seguridad se descargará e instalará automáticamente. Los clientes que no han habilitado la actualización automática deben buscar las actualizaciones e instalar esta actualización manualmente. Para obtener información sobre las opciones de configuración específicas de la actualización automática, vea el [artículo 294871 de Microsoft Knowledge Base](http://support.microsoft.com/kb/294871).

Para administradores e instalaciones empresariales, o usuarios finales que deseen instalar esta actualización de seguridad manualmente, Microsoft recomienda que los clientes apliquen la actualización inmediatamente con el software de administración de actualizaciones o busquen las actualizaciones con el servicio [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747).

Vea también la sección **Herramientas y consejos para la detección e implementación**, más adelante en este boletín.

**Problemas conocidos.** En el [artículo 960803 de Microsoft Knowledge Base](http://support.microsoft.com/kb/960803) se señalan los problemas detectados que pueden experimentar los clientes al instalar esta actualización de seguridad. Se documentan asimismo las soluciones propuestas.

**¿Contiene esta actualización algún cambio en relación con la seguridad en su funcionalidad?**  
Sí, para resolver el problema descrito en CVE-2009-0089, Microsoft ha implementado una nueva clave del Registro UseFQDNFallbackForCnCompare en SOFTWARE\\Microsoft\\Windows\\CurrentVersion\\Internet Settings\\WinHttp, que controla el comportamiento de WinHTTP al validar el nombre de dominio completo de un host según la entrada de nombre distintivo de un certificado.

De forma predeterminada, esta clave del Registro no existe, lo que garantiza que WinHTTP valida por completo que el nombre distintivo (DN) incluido en el certificado es idéntico al DN con el que se ha iniciado la conexión. El comportamiento anterior se puede volver a habilitar si esta clave del Registro se establece en True. En este caso, la validación será menos estricta y se aceptará si el certificado coincide con un cambio de nombre a través del reenvío en el nivel DNS. Se trata del comportamiento menos seguro que es el predeterminado antes de la instalación de esta actualización de seguridad.

Vulnerabilidad de subdesbordamiento de enteros en los servicios HTTP de Windows (CVE-2009-0086)  
-----------------------------------------------------------------------------------------------
  
<span></span>
Existe una vulnerabilidad de ejecución remota de código en la forma en que los servicios HTTP de Windows tratan los valores específicos que devuelve un servidor web. Un atacante que aprovechara esta vulnerabilidad podría lograr el control completo de un sistema afectado. De esta forma, un atacante podría instalar programas; ver, cambiar o eliminar datos; o crear cuentas nuevas con todos los derechos de usuario como el servicio o la aplicación que llama a la API WinHTTP para conectarse al sitio web del atacante.
  
Para ver esta vulnerabilidad como una entrada estándar en la lista de vulnerabilidades y exposiciones comunes, consulte [CVE-2009-0086](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0086).
  
#### Factores atenuantes para la vulnerabilidad de subdesbordamiento de enteros en los servicios HTTP de Windows (CVE-2009-0086)
  
El factor atenuante hace referencia a una opción, configuración común o práctica recomendada general, existente en un estado predeterminado, que podría reducir la gravedad de una vulnerabilidad. Los siguientes factores atenuantes podrían ser útiles en su situación:
  
-   Todas las ediciones compatibles de Internet Explorer no están afectadas por esta vulnerabilidad porque el software no usa los servicios HTTP de Windows.
  
#### Soluciones provisionales para la vulnerabilidad de subdesbordamiento de enteros en los servicios HTTP de Windows (CVE-2009-0086)
  
Microsoft no ha identificado soluciones provisionales para esta vulnerabilidad.
  
#### Preguntas más frecuentes sobre la vulnerabilidad de subdesbordamiento de enteros en los servicios HTTP de Windows (CVE-2009-0086)
  
**¿Cuál es el alcance de esta vulnerabilidad?**    
Se trata de una vulnerabilidad de ejecución remota de código. Un atacante que aprovechara esta vulnerabilidad podría lograr el control completo de un sistema afectado. De esta forma, un atacante podría instalar programas; ver, cambiar o eliminar datos; o crear cuentas nuevas con todos los derechos de usuario como el servicio o la aplicación que llama a la API WinHTTP para conectarse al sitio web del atacante.
  
Esta vulnerabilidad afecta a los componentes de Windows y las aplicaciones de terceros que usan los servicios HTTP de Windows y están configurados para conectarse a servidores web remotos y que no son de confianza.
  
**¿Cuál es la causa de esta vulnerabilidad?**    
La vulnerabilidad se debe a la ausencia de validación de un parámetro devuelto por un servidor web remoto. Esto puede provocar la ejecución arbitraria de código.
  
**¿Qué son los servicios HTTP de Windows?**    
Los servicios HTTP de Windows (WinHTTP) proporcionan a los desarrolladores una interfaz de programación de aplicaciones (API) cliente HTTP para enviar solicitudes a través del protocolo HTTP a los servidores web. Los componentes de Microsoft Windows y el software de terceros pueden usar WinHTTP. El servicio Plug and Play universal (UPnP), entre otros, usa WinHTTP.
  
**¿Qué es Plug and Play universal?**    
Plug and Play universal (UPnP) es una capacidad que permite a los dispositivos de una red detectar otros dispositivos y determinar el modo de funcionamiento con ellos. El concepto de UPnP se entiende más fácilmente si se compara con la capacidad plug-and-play (PnP) con la que está familiarizada la mayoría de usuarios de Windows. PnP permite al sistema operativo detectar componentes de hardware nuevos cuando se instalan en el sistema. Por ejemplo, cuando se instala un nuevo mouse en el equipo, PnP permite que Windows lo detecte, cargue los controladores necesarios y empiece a usarlo. UPnP extiende este concepto a los dispositivos de una red, en vez de hacerlo en el mismo sistema local. UPnP permite que los equipos conozcan otros dispositivos de la red y determinen cómo usarlos. Por ejemplo un equipo podría usar UPnP para detectar si hay impresoras en la red que pueda usar y saber cómo usarlas.
  
**¿Qué es el protocolo simple de detección de servicios (SSDP)?**    
La detección de dispositivos es el proceso por el que los equipos con capacidad UPnP tienen constancia de la disponibilidad de dispositivos que pueden usar y saber cómo solicitarles servicios. Cuando arranca un equipo con capacidad UPnP, ya puede haber dispositivos en la red que lo puedan usar. Para determinar si es así, el equipo envía una solicitud de difusión (denominada directiva M-SEARCH) en la que pide que cualquier dispositivo con capacidad UPnP dentro de la subred responda a ella y proporcione información acera de cómo usarlo.
  
De forma parecida, cuando arranca un dispositivo que admite UPnP (por ejemplo, una impresora con capacidad UPnP), ya puede haber equipos con capacidad UPnP en la red que podrían usarla. El dispositivo difunde un mensaje (denominado directiva NOTIFY) a todos los equipos dentro de la subred para anunciar su presencia en la red e invitar a los equipos a usar sus servicios.
  
**¿Para qué puede usar un atacante esta vulnerabilidad?**    
Un atacante que aprovechara esta vulnerabilidad podría obtener los mismos derechos de usuario que el servicio o la aplicación que llama a la API WinHTTP para conectarse al servidor web del atacante. Las aplicaciones que se ejecutan en cuentas que están configuradas con pocos derechos de usuario en el sistema están menos afectadas que las que se ejecutan en cuentas con derechos de usuario administrativos.
  
**¿Cómo podría aprovechar un atacante la vulnerabilidad?**    
Un atacante debe obligar o atraer a un programa que use los servicios HTTP de Windows para que se conecte a su servidor web malintencionado. Es importante tener en cuenta que el servicio Plug and Play universal (UPnP) usa las bibliotecas de WinHTTP. Si este servicio está deshabilitado, los usuarios malintencionados en la subred local pueden responder a las solicitudes SSDP y provocar que el servicio UPnP se conecte mediante WinHTTP a un host malintencionado que podría aprovechar esta vulnerabilidad. El servicio UPNP se ejecuta en la cuenta LocalService, que es una cuenta con privilegios limitados en el sistema.
  
**¿Qué sistemas están más expuestos a esta vulnerabilidad?**    
Los sistemas con más riesgo de sufrir un ataque son las estaciones de trabajo y los servidores de Servicios de Terminal Server. Los servidores podrían correr un mayor riesgo si los administradores permiten a los usuarios iniciar sesión en los servidores y ejecutar programas. Sin embargo, las prácticas recomendadas desaconsejan permitir esto.
  
**¿Cómo funciona esta actualización?**    
La actualización elimina la vulnerabilidad al modificar la manera en que los servicios HTTP de Windows tratan el error que da lugar a la situación vulnerable.

#### Preguntas más frecuentes sobre la vulnerabilidad de error de coincidencia de nombre de certificado en los servicios HTTP de Windows (CVE-2009-0089)
  
**¿Cuál es el alcance de esta vulnerabilidad?**    
Se trata de una vulnerabilidad de suplantación de identidad que existe en los servicios HTTP de Windows. Cuando se combina con otros ataques específicos, como la suplantación de DNS, puede permitir que el atacante consiga suplantar el certificado digital de un sitio web para cualquier aplicación que use los servicios HTTP de Windows.
  
**¿Cuál es la causa de esta vulnerabilidad?**    
Cuando una aplicación llama a los servicios HTTP de Windows para configurar una conexión a un servidor web remoto, WinHTTP sólo valida el nombre de dominio completo de la dirección URL según el certificado del sitio web original al que intenta conectarse el host. Las combinaciones específicas de suplantación de DNS pueden reenviar una conexión a otro servidor web que tenga un certificado válido para el host específico, pero no para la página web con la que WinHTTP ha iniciado la conexión. La combinación de suplantación de DNS y esta vulnerabilidad podría provocar que WinHTTP considerara como aceptable el certificado del servidor web remoto. Como WinHTTP depende de la aplicación para la interacción del usuario, puede que la aplicación no advierta al usuario de esta discrepancia.
  
**¿Qué son los servicios HTTP de Windows?**    
Los servicios HTTP de Windows (WinHTTP) proporcionan a los desarrolladores una interfaz de programación de aplicaciones (API) cliente HTTP para enviar solicitudes a través del protocolo HTTP a otros servidores HTTP.
  
**¿Qué son los certificados digitales?**    
El certificado digital es una credencial común que se usa para autenticar la identidad. Una organización de confianza asigna un certificado a un individuo o una entidad, el sujeto de ese certificado. La organización de confianza que emite el certificado es una entidad de certificación (CA) y se denomina el emisor del certificado. Una entidad de certificación emitirá un certificado sólo después de comprobar la identidad del sujeto del certificado.
  
**¿Qué es la suplantación de DNS?**    
El sistema de nombres de dominio (DNS) forma parte del conjunto estándar de protocolos que componen TCP/IP. El DNS se implementa mediante dos componentes de software: servidor DNS y cliente DNS (resolución). Ambos componentes se ejecutan como aplicaciones de servicio en segundo plano. Los recursos de red se identifican mediante direcciones IP numéricas, pero dichas direcciones resultan difíciles de recordar por parte de los usuarios de red. La base de datos DNS contiene registros que asignan nombres alfanuméricos descriptivos para los recursos de red a las direcciones IP que usan dichos recursos para la comunicación. De este modo, DNS actúa como un dispositivo nemotécnico, que permite a los usuarios de red recordar más fácilmente los recursos de red. Para obtener más información y ver diagramas lógicos que ilustren el modo en que DNS se adapta a otras tecnologías de Windows, consulte el artículo de TechNet [¿Qué es DNS?](http://technet.microsoft.com/en-us/library/cc787921.aspx)
  
La suplantación de DNS es una categoría amplia de ataques en los que en un punto del sistema DNS se incorporan respuestas DNS incorrecta por parte de un tercero que normalmente no se ocupa de estas respuestas. Un ataque de este tipo puede provocar que un nombre de dominio o de host se resuelva en una dirección IP cuya resolución no estaba prevista por los propietarios de dicho nombre. En tal caso, se puede pedir a un host que se conecte a un servidor de terceros potencialmente malintencionado.
  
**¿Para qué puede usar un atacante esta vulnerabilidad?**    
Un atacante podría suplantar un sitio web seguro y ofrecer contenido malintencionado a la aplicación mediante los servicios HTTP de Windows, que confiarían en él como su fiera un sitio web seguro.
  
**¿Cómo podría aprovechar un atacante la vulnerabilidad?**    
Un atacante debe combinar esta vulnerabilidad con la suplantación de DNS o un ataque similar para dirigir la conexión saliente a un servidor bajo su control. A continuación, podría ofrecer un certificado válido para el sitio web al que se ha conectado, que los servicios HTTP de Windows validarían incorrectamente como válido para el sitio web host de destino al que el usuario intentó conectarse inicialmente.
  
**¿Qué sistemas están más expuestos a esta vulnerabilidad?**    
Los sistemas con más riesgo de sufrir un ataque son las estaciones de trabajo y los servidores de Servicios de Terminal Server. Los servidores podrían correr un mayor riesgo si los administradores permiten a los usuarios iniciar sesión en los servidores y ejecutar programas. Sin embargo, las prácticas recomendadas desaconsejan permitir esto.
  
**¿Cómo funciona esta actualización?**    
La actualización elimina la vulnerabilidad al cambiar la forma en que los servicios HTTP de Windows validan los certificados que pertenecen a un sitio web seguro.

Vulnerabilidad de reflexión de certificados en los servicios HTTP de Windows (CVE-2009-0550)  
--------------------------------------------------------------------------------------------
  
<span></span>
Existe una vulnerabilidad de ejecución remota de código en la forma en que los servicios HTTP de Windows tratan las credenciales NTLM cuando un usuario se conecta al sitio web de un atacante. Esta vulnerabilidad permite a un atacante repetir las credenciales del usuario y ejecutar código en el contexto del usuario que ha iniciado sesión. Si un usuario inicia sesión con derechos de usuario administrativos, un intruso que aprovechara esta vulnerabilidad podría lograr el control completo de un sistema afectado. De esta forma, un intruso podría instalar programas; ver, cambiar o eliminar datos; o crear cuentas nuevas con todos los derechos de usuario. Por tanto, los usuarios cuyas cuentas estén configuradas con pocos derechos de usuario en el sistema correrían un riesgo menor que aquellos que cuenten con derechos de usuario administrativos.
  
Para ver esta vulnerabilidad como una entrada estándar en la lista de vulnerabilidades y exposiciones comunes, consulte [CVE-2009-0550](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0550).
  
#### Factores atenuantes para la vulnerabilidad de reflexión de certificados en los servicios HTTP de Windows (CVE-2009-0550)
  
El factor atenuante hace referencia a una opción, configuración común o práctica recomendada general, existente en un estado predeterminado, que podría reducir la gravedad de una vulnerabilidad. Los siguientes factores atenuantes podrían ser útiles en su situación:
  
-   En el caso de un ataque basado en web, el intruso podría alojar un sitio web que contuviera una página web para aprovechar esta vulnerabilidad. Además, los sitios web vulnerables y los sitios web que aceptan o alojan contenido o anuncios proporcionados por el usuario podrían incluir contenido especialmente diseñado que permita aprovechar esta vulnerabilidad. Sin embargo, el atacante no podría en ningún caso obligar a los usuarios a visitar estos sitios web. Por lo tanto, tendría que atraerlos al sitio web; por lo general, convenciéndoles para que hagan clic en un vínculo de un mensaje de correo electrónico o de Instant Messenger que lleve a los usuarios al sitio web del atacante.  
-   Un intruso que aprovechara esta vulnerabilidad podría conseguir el mismo nivel de derechos de usuario que el usuario local. Por tanto, los usuarios cuyas cuentas estén configuradas con pocos derechos de usuario en el sistema correrían un riesgo menor que aquellos que cuenten con derechos de usuario administrativos.  
-   De forma predeterminada, todas las ediciones compatibles de Windows Vista y Windows Server 2008 tienen el perfil para el firewall de Windows configurado en público, lo que reduce la exposición en estas plataformas a la vulnerabilidad porque con el perfil público el firewall bloquea el tráfico SMB. Para que se realice el ataque y se reflejen las credenciales, la vía de ataque más habitual es SMB, ya que el cliente necesita ejecutar un servicio que acepte autenticación NTLM.

Un atacante tendría que hospedar un servidor web especialmente diseñado que capture las credenciales de inicio de sesión NTLM del usuario. -   **Microsoft Baseline Security Analyzer**

    Para comprobar que una actualización de seguridad se ha aplicado en un sistema afectado, puede tratar de utilizar la herramienta Baseline Security Analyzer (MBSA) de Microsoft. Consulte la sección **Herramientas y consejos de detección e implementación**, anteriormente en este boletín, para obtener más información.

-   **Comprobación de la versión del archivo**

    Puesto que existen varias ediciones de Microsoft Windows, los pasos que se indican a continuación pueden ser diferentes en su sistema. En tal caso, consulte la documentación del producto para ejecutar los pasos necesarios.

    1.  Haga clic en **Inicio** y, después, en **Buscar**.
    2.  En el panel **Resultados de la búsqueda**, haga clic en **Todos los archivos y carpetas** del **Asistente para búsqueda**.
    3.  En el cuadro **Todo o parte del nombre de archivo**, escriba un nombre de archivo de la tabla de información de archivos correspondiente y, a continuación, haga clic en **Buscar**.
    4.  En la lista de archivos, haga clic con el botón secundario del mouse en un nombre de archivo de la tabla de información de archivos correspondiente y, después, haga clic en **Propiedades**.
        **Nota** Según la edición del sistema operativo o de los programas instalados, cabe la posibilidad de que algunos de los archivos enumerados en la tabla de información de archivos no estén instalados.
    5.  En la ficha **Versión**, determine la versión del archivo instalado en el sistema por comparación con la versión documentada en la tabla de información de archivos correspondiente.
        **Nota** Los atributos distintos a la versión del archivo pueden cambiar durante la instalación. La comparación de otros atributos de los archivos con los datos de la tabla de información de archivos no constituye un método apropiado para comprobar que se haya aplicado la actualización. Además, en ciertos casos, el nombre de los archivos se puede cambiar durante la instalación. Si no aparece información de la versión o del archivo, recurra a otro de los métodos disponibles para comprobar la instalación de la actualización.

-   **Comprobación de las claves del Registro**

    También puede comprobar los archivos que ha instalado esta actualización de seguridad si consulta las claves del Registro enumeradas en **Tabla de referencias** de esta sección.

    Estas claves del Registro pueden no contener una lista completa de archivos instalados. Además, estas claves del Registro pueden no crearse correctamente si el administrador o un OEM integran o incorporan esta actualización de seguridad en los archivos fuente de instalación de Windows.

Esta actualización de seguridad admite los parámetros de instalación que se indican a continuación.
  
| Parámetro         | Descripción                                                                                                                                               |  
|-------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------|  
| **/?, /h, /help** | Muestra ayuda sobre los parámetros admitidos.                                                                                                             |  
| **/quiet**        | Suprime la pantalla de estado o los mensajes de error.                                                                                                    |  
| **/norestart**    | Cuando se combina con **/quiet**, el sistema no se reiniciará después de la instalación, incluso si se necesita un reinicio para terminar la instalación. |
  
**Nota** Para obtener más información acerca del programa de instalación wusa.exe, consulte el [artículo 934307 de Microsoft Knowledge Base](http://support.microsoft.com/kb/934307).
  
**Comprobación de que se ha aplicado la actualización**
  
-   **Microsoft Baseline Security Analyzer**
  
    Para comprobar que una actualización de seguridad se ha aplicado en un sistema afectado, puede tratar de utilizar la herramienta Baseline Security Analyzer (MBSA) de Microsoft. Consulte la sección **Herramientas y consejos de detección e implementación**, anteriormente en este boletín, para obtener más información.
  
-   **Comprobación de la versión del archivo**
  
    Puesto que existen varias ediciones de Microsoft Windows, los pasos que se indican a continuación pueden ser diferentes en su sistema. En tal caso, consulte la documentación del producto para ejecutar los pasos necesarios.
  
    1.  Haga clic en **Inicio** y, a continuación, escriba un nombre de archivo de actualización en **Iniciar búsqueda**.  
    2.  Cuando el archivo aparezca en **Programas**, haga clic con el botón secundario en el nombre de archivo y haga clic en **Propiedades**.  
    3.  En la ficha **General**, compare el tamaño de archivo con las tablas de información de archivo que se han proporcionado en el artículo de KB del boletín.  
    4.  También puede hacer clic en la ficha **Detalles** y comparar la información, como la versión de archivo y la fecha de modificación, con las tablas de información de archivo que se han proporcionado en el artículo de KB del boletín.  
    5.  Finalmente, también puede hacer clic en la ficha **Versiones anteriores** y comparar la información de archivo de la versión anterior del archivo con la información de archivo de la versión nueva o actualizada del archivo.
